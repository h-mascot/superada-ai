I run five AI agents. Ada, Scotty, Spock, Book, and Zora. They live on different machines - a Linux gateway, a Raspberry Pi, a VM, and a Mac Mini. Each one has a job. Each one needs to pull tasks from a shared Mission Control board, do the work, and report back.

For months, this was held together with manually copied shell scripts and hope.

Yesterday I decided to fix that. What followed was a 14-hour debugging marathon that produced 15 versions of a deployment system, exposed every fragile assumption in the stack, and ended with all five agents autonomously clearing a 23-task backlog overnight while I slept.

Here's what actually happened.

## The starting point: scripts on a clipboard

Mission Control is a task board backed by SQLite and a REST API. Agents interact with it through shell scripts: `mc.sh` for lifecycle operations (create, start, review, done), `mc-auto-pull.sh` for grabbing the next available task, `mc-stall-check.sh` for flagging stuck work, and a few others.

The problem was deployment. When Zora needed these scripts on the Mac, I'd SSH in, copy the files, edit the paths, add a cron entry, and pray nothing drifted. When I fixed a bug in `mc.sh`, I'd fix it locally and forget that four other agents were running the old version. No versioning. No verification. No rollback path.

The first question was simple: should this be one skill or five? A council of perspectives (systems architect, reliability engineer, pragmatic engineer, operator) all converged on the same answer: one thin bootstrap skill with a shared runtime bundle and per-agent manifests. Don't over-abstract. Don't build a package manager. Just make the install repeatable.

## v1: It works on my machine

The first version was straightforward. A skill directory at `skills/entity-mc/` containing:

- `install.sh` - stages a release, activates it, installs wrappers and cron
- `verify.sh` - checks everything landed correctly
- `rollback.sh` - reverts to the previous known-good version
- `lib.sh` - shared functions for rendering cron blocks, staging releases
- Per-agent manifests defining target paths, cron schedules, and agent names

Install stages a release under `.entity-mc/releases/<version>`, symlinks `.entity-mc/runtime` to it, drops wrapper scripts in the agent's `scripts/` directory, and adds a cron block. Verify checks the state directory exists, the version matches, all scripts are executable, and the cron markers are present.

Tested it in a temp directory. `INSTALL_OK`. `VERIFY_OK`. Shipped it to all four target agents. Done, right?

## v2: The review validator was too dumb

While packaging the scripts, I noticed `mc.sh`'s review validation was weak. It checked that output existed, was over 50 characters, and wasn't literally "done" or "n/a". But it let through garbage like "Full analysis in subagent output" with no file path, no link, no actual evidence. One of our eval tasks (#222, TinyClaw evaluation) had been sitting in review for weeks with exactly that kind of non-output.

Patched both the client-side `mc.sh` and the server-side Task Master hooks to reject vague handwave phrases. The server already had a `review_check` event that ran when tasks entered review - it just wasn't checking for semantic quality. Added rejection patterns for invisible references ("see notes", "shared above", "full report in agent output") and required research/eval tasks to include actual file paths or URLs.

Bumped to v2, redeployed. This is where the fun started.

## v3: The cron lines were smashed together

First real deployment bug. The cron renderer in `lib.sh` was putting both the auto-pull and stall-check entries on a single line. Instead of two separate cron entries, agents got something like:

```
*/30 * * * * ... mc-auto-pull.sh >> cron.log 2>&10 */2 * * * ... mc-stall-check.sh
```

That `2>&10` is two entries concatenated. The cron daemon silently ignored it. Zero auto-pulls had actually fired on any agent since the initial rollout.

Fixed the heredoc rendering, hardened `verify.sh` to validate cron line correctness (not just marker presence), bumped to v3.

## v4: Mac runs Bash 3

Deployed v3, ran a manual test on Book (Mac). Immediate crash:

```
declare: -A: invalid option
```

`mc-auto-pull.sh` uses associative arrays (`declare -A AGENT_MODELS`). macOS ships Bash 3.2 from 2007. Associative arrays need Bash 4+. Homebrew installs Bash 5 at `/opt/homebrew/bin/bash`, but the cron wasn't using it.

Updated the Mac manifests to explicitly set `ENTITY_MC_BASH_BIN=/opt/homebrew/bin/bash` and patched the cron renderer to use it. Bumped to v4.

## v5-v6: The wrapper recursion infinite loop

This one was subtle and took a while to find.

The installer copies scripts from a source directory into a release. But on agents that had already been installed, the `scripts/` directory contained wrapper stubs from the previous install - not the real script bodies. The wrapper stubs look like this:

```bash
#!/bin/bash
exec /path/to/.entity-mc/runtime/mc-auto-pull.sh "$@"
```

The staging function was copying these wrappers into the release. Then the runtime symlinked to the release. So the wrapper called the runtime, which was the release, which was a wrapper, which called the runtime...

Infinite exec recursion. 100% CPU. The processes never terminated.

Fix: added a `source-scripts/` directory inside the skill bundle as the canonical source. The installer now stages from there, never from the target's existing `scripts/` directory. Added a guard that refuses to stage a file if it looks like a wrapper stub.

## v7: MAX_DOING was 2

Auto-pull had a hardcoded `MAX_DOING=2`. Ada had 3 tasks stuck in doing from earlier manual work. Every single cron run: `{"action":"skip","reason":"at_capacity"}`. For hours.

Changed it to 10. Immediately pulled a task. Sometimes the fix is one character.

## v8: Auto-pull doesn't execute anything

This was the architectural gap nobody noticed until everything else was working. Auto-pull moves a task from `todo` to `doing` on the board. That's it. It doesn't start an agent session. It doesn't spawn a process. The task just sits in `doing` until someone manually notices.

Added an execution step: after pulling a task, the script spawns `openclaw agent` (or `hermes chat` on Book's Hermes runtime) with a built prompt containing the task details, context, and instructions to call `mc.sh review` when done.

But Book and Zora are on Mac, where `openclaw` isn't in the default PATH. And Book runs Hermes, not OpenClaw. So the execution bridge needed runtime detection with fallback binary discovery across `/opt/homebrew/bin/`, `$HOME/.local/bin/`, and `/usr/local/bin/`.

## v9: Hermes doesn't speak OpenClaw

Book runs on the Hermes framework. Different CLI, different invocation pattern. `hermes chat -q "prompt" --yolo` instead of `openclaw agent -m "prompt"`.

Patched the execution step to check `ENTITY_MC_RUNTIME` from the manifest and branch accordingly. Hermes agents get `hermes chat -q`, OpenClaw agents get `openclaw agent -m`. Both write to `exec.log` and emit JSON with the spawned PID.

Tested Book pulling task #361 and Zora pulling #312. Both spawned real processes. First time all five agents could both pull AND execute.

## v10-v11: Context enrichment and smart pulling

The agents were executing, but they were flying blind. The spawn prompt was just the raw task description. No memory files, no tools reference, no project context.

Wired `mc-build-context.sh` into the execution path. It assembles a context preamble from memory files, safety rules, credential locations, skill guidance, and auto-inferred project context based on task keywords. An entity task gets entity context files. A superada task gets blog publishing context.

Also added smart second-pull logic: if the first pulled task has an estimate under 1 hour, the script automatically pulls a second task in the same run. Changed cron from every 30 minutes to every 10.

Then discovered Mac agents couldn't load context because `mc-build-context.sh` hardcodes `$HOME/clawd/memory/` paths that don't exist on Mac. Set up an hourly rsync from the gateway to Mac to keep memory files in sync.

## v12: Per-agent and per-task prompt injection

Added two prompt layers on top of the built context:

1. A `spawn-prompt.md` file in each agent's state directory - loaded on every spawn
2. A `prompt` field in task metadata - loaded only for that specific task

This lets you customize agent behavior ("always search memory before escalating") and task behavior ("focus on pricing comparison, output a markdown table") without touching the core scripts.

## v13: The watchdog and the exit contract

Agents were pulling and executing, but some sessions would die without calling `mc.sh review`. The task would rot in `doing` forever.

Added a watchdog that runs before each pull attempt. It scans for previous spawn records, checks if the process is still alive, and if it's dead with the task still in `doing`, moves the task back to `todo` with a comment explaining what happened. If a process has been running over 45 minutes, it kills it. The entire watchdog has a hard 30-second timeout so it can't loop itself.

Also strengthened the spawn prompt with an exit contract at both the top and bottom: "You MUST run `mc.sh review` when done. No exceptions. If you skip this, the task is orphaned."

## v14-v15: Blocker escalation

The last piece. When an agent hits a wall - missing credentials, access denied, unclear requirements - what should it do?

Added a blocker protocol to the spawn prompt. Before escalating, agents search all memory files, secrets, and .env files for answers. If the blocker is real and unresolvable, they post to a Discord escalation channel with the task number, what they tried, the exact blocker, and what they need from me to unblock. Then they mark the task blocked and move it back to todo so it doesn't rot.

Three safety layers now: the prompt says "you must close the loop", the blocker protocol gives agents an escape hatch that notifies me, and the watchdog catches anything that falls through both.

## The overnight results

Deployed v15 at 22:33 UTC. Assigned 23 tasks across agents: simple tasks to Book, research to Spock, coding to Zora and Ada.

Went to bed.

Woke up to this:

- **Todo: 0** (was 23)
- **Review: 51** (was 14)
- **All five agents had pulled, executed, and closed tasks**

Spock got excited and spawned extra research tasks during execution. Book cleared 8 items. The system was self-sustaining.

## What I'd do differently

The version count tells you something. Fifteen deploys in one day means I was fixing things I should have caught in design. The wrapper recursion bug (v5-v6) was entirely preventable if I'd thought about "what's already in the target directory?" The Bash 3 issue on Mac is a classic that should be in every deployment checklist. The cron concatenation bug was a heredoc formatting mistake that `verify.sh` should have caught from v1.

But here's the thing: I learned more about my agent infrastructure in this one day than in the previous month of running it. Every version exposed an assumption. Every fix made the system more honest about what it actually needs to work.

The conveyor belt pattern - pull, enrich, execute, verify, escalate or close - is simple. Getting there required touching cron rendering, Bash compatibility, process locking, runtime detection, context assembly, prompt engineering, watchdog timeouts, and cross-host file synchronization. None of those are hard individually. But they interact, and the interactions are where systems actually break.

The agents are still running. The cron fires every 10 minutes. When it finds work, it pulls it, builds context, spawns a session, and moves on. When it doesn't, it says so in one line of JSON and exits. When something goes wrong, it tells me on Discord instead of dying quietly.

Fifteen versions. One day. Zero manual task assignments needed going forward.

That feels about right for v1 of anything real.
