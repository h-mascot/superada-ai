Channel/Product Fit

  • My motivation is specifying different role/behavior per whatsapp group. Also address https://github.com/openclaw/openclaw/issues/7011
  • Hey, tracked down the root cause of the silent channel death after timeouts — opened a fix. PR: [#60015](https://github.com/openclaw/openclaw/pull/60015) Issue: [#59983](https://github.com/openclaw/openclaw/issues/59983) Root cause: The `finally` block in `src/agents/pi-embedded-runner/run/attempt.ts` executes several async cleanup steps sequentially without individual error guards. If any of them (like `flushPendingToolResultsAfterIdle()`) throws during an abort/timeout, the block short-circuits and `sessionLock.release()` is never reached. The lock leaks. Worse, when the process restarts, the gateway hits the orphaned `.lock` file on disk. The secondary defense layer (`shouldTreatAsOrphanSelfLock`) had a bug on Linux where it unconditionally returned `false` if a `starttime` was present, and on macOS it couldn't verify `starttime` at all. So the orphan lock is never reclaimed, the gateway times out waiting for it, and the message is silently dropped deep in the routing layer. That's why channels appear permanently dead with zero logs. Fix: - Wrapped each cleanup step in the `finally` block with its own `try/catch` so `sessionLock.release()` is guaranteed to execute, eliminating the root cause of the leak. - Fixed the orphan detection logic in `shouldTreatAsOrphanSelfLock` to correctly identify same-process orphans on Linux via `starttime` comparison. - Added a 5-second age guard to prevent race conditions during concurrent lock acquisitions. - Tests covering the above. Fixes [#59983](https://github.com/openclaw/openclaw/issues/59983) — would appreciate a review when someone gets a chance, thanks!
